# 面试常见问答

## 1. 为什么不直接做单 Agent RAG？

因为导购问题并不都是简单问答。

- 单事实题可以直接检索回答
- 比较题需要先取值再计算
- 推荐题需要先召回候选，再排序，再给理由
- 聚合题更适合走 SQL

如果全部交给单 Agent RAG，自由生成会太多，稳定性差，也不容易解释。

## 2. 为什么比较题要走 `retrieval -> extraction -> calculation -> response`？

因为比较题不是直接摘抄型问题，而是一个小型工作流。

例如:
`A 和 B 哪个更贵，差多少元？`

至少要完成:

1. 找到 A 的证据
2. 从证据里抽出 A 的价格
3. 找到 B 的证据
4. 从证据里抽出 B 的价格
5. 计算差值
6. 输出结论

把这几个步骤拆开后，结果更稳定，也更好测试。

## 3. 为什么要单独拆 `extraction_agent`？

因为检索和抽取是两类不同能力。

- `retrieval_agent` 解决的是“把相关证据找回来”
- `extraction_agent` 解决的是“把证据变成结构化字段和指标”

如果把这两件事混在一起，retrieval worker 会过厚，推荐题和比较题也不够清晰。拆开以后，链路更像标准工作流，也更容易调试。

## 4. recommendation_agent 的作用是什么？

它负责把“推荐”从普通检索变成有约束的筛选和排序。

推荐题不是只找相关文本，而是要结合:

- 预算
- 品类
- 用户场景
- 商品亮点

然后给出候选和推荐理由。

## 5. 为什么推荐不能只看预算？

因为同样在预算内的商品，适合的场景可能完全不同。

比如办公更看重续航和轻薄，游戏更看重性能和散热。

所以我给 recommendation worker 加了轻量场景权重，让推荐更像导购决策，而不是简单价格过滤。

## 6. 为什么一个 Agent 会有多个工具，但不一定每次全调用？

因为 Agent 的价值不在于“固定跑完所有工具”，而在于“能在自己的领域里选择合适的工具”。

例如:

- `retrieval_agent` 默认先 `hybrid_search`，候选多时再 `rerank_tool`
- `extraction_agent` 默认先 `structured_extractor`，结果需要清洗时再 `field_normalizer`
- `recommendation_agent` 默认先 `catalog_filter`，再用 `candidate_ranker`
- `sql_agent` 先 `sql_guard`，再决定执行还是只解释

这样比“每个 Agent 只绑一个工具然后固定调用”更像真正的 specialist agent。

## 7. Judge 和 Replanner 解决了什么问题？

它们解决的是“多步骤执行不稳定”的问题。

Judge 会检查每一步是否满足成功条件，比如:

- 检索有没有拿到目标字段
- 计算有没有产出结果
- 推荐有没有候选

如果没有满足，就进入 Replanner，做一次重试或切换策略，比如:

- 扩展检索 query 重试
- 从 retrieval 切到 search 补证据

## 8. 为什么 Response 要优先读结构化 artifacts？

因为前面的 Worker 已经做了检索、计算、推荐排序这些确定性工作。

如果最后还是完全靠 LLM 自由生成，就容易把中间结果重新“猜错”。

所以 Response 优先读结构化 artifacts，只有结构化信息不足时才回退到自由生成。

## 9. 这个项目最想体现什么能力？

不是单点 API 调用，而是：

- 问题建模
- 多 Agent 协作设计
- 结构化执行流程
- 稳定性控制
- 面向业务场景的系统拆解能力

## 10. 现在 6 个 Worker Agent 合理吗？

合理。

因为当前业务能力正好对应 6 类核心任务:

- 检索
- 结构化抽取
- 推荐
- 计算
- SQL 分析
- 外部搜索

这个数量已经足够，不需要为了“更多 Agent”继续硬拆。后续重点应该放在工具选择逻辑、测试覆盖和执行链路可视化上。

## 11. 如果继续迭代，你会做什么？

- 增强 recommendation worker 的候选召回与排序
- 提升 replanner 的动态重规划能力
- 增加更多导购测试集
- 增加执行链路可视化
